# RLS & Patient Data Isolation Audit — 2026-05-08

**Audit scope:** Every table in the `public` schema, every Supabase
edge function under `supabase/functions/`, and five patient-isolation
test scenarios.

**Method:** Migration-replay analysis (79 migrations replayed in
lexicographic order to determine live policy state) + per-edge-function
JWT/service-role/role-check trace + scenario reasoning. **No SQL was
run against the live database**; all findings are derived from the
codebase. Live execution should be run as a follow-up by a human with
read-only DB access.

**Status:** This report identifies findings only. No policy changes
were made. RLS changes need careful review before deployment because
mistakes are silent and exfiltrate PHI.

## 1. Helper functions referenced in RLS policies

### `public.has_role(_user_id uuid, _role app_role) returns boolean`
```sql
SELECT EXISTS (
  SELECT 1 FROM public.user_roles
  WHERE user_id = _user_id AND role = _role
)
```
Returns true when the user has exactly that `app_role`. The enum has
evolved over time; current values: `admin`, `staff`, `user`,
`business_admin`, `provider`.

### `public.has_business_admin_role(_user_id uuid) returns boolean`
```sql
SELECT EXISTS (
  SELECT 1 FROM public.user_roles
  WHERE user_id = _user_id AND role = 'business_admin'::app_role
)
```
Used by a small number of finance/operations policies.

**Anomaly:** No `is_staff()` or `is_admin()` helper functions exist.
Several policies use the literal pattern
`has_role(auth.uid(), 'admin') OR has_role(auth.uid(), 'staff')`,
which is duplicated dozens of times. A single `is_clinical_staff()`
helper would tighten consistency and reduce the chance of one policy
drifting from the rest.

## 5. Findings & recommended remediations (do not apply without review)

### F-1 — `iv_drip_bookings` SELECT is fully public — CRITICAL

> **Status (2026-05-09):** Remediation written, awaiting Lovable apply.
> See `supabase/migrations/20260510030000_critical_rls_fixes.sql`. The
> migration drops the `USING (true)` policy, adds a patient-self-read
> policy keyed on customer_email or appointment chain, and exposes
> `get_iv_booking_by_stripe_session(text)` as the sanctioned anon read
> path for the post-payment surface. The IV post-payment frontend
> (`IVPaymentSuccess.tsx`) does not currently read this table directly,
> so the policy drop is non-breaking.

**File:** `supabase/migrations/20260507171224_4973338c-c6ff-4b53-98f3-69f8d70c56f7.sql:82-83`

**Current:**
```sql
CREATE POLICY "Public can view by stripe session id"
ON public.iv_drip_bookings FOR SELECT USING (true);
```

**Problem:** `USING (true)` means every row is visible to every caller including anonymous. Policy name implies session-id scoping but the clause doesn't enforce it. Patient PII (name, email, phone), scheduled time, Stripe session IDs all leak.

**Why it exists:** Probably to let the post-checkout confirmation page (running unauthenticated) read its own booking back by session ID.

**Recommended fix (R-1):**
Drop the policy and route the post-checkout read through an edge function `verify-iv-payment` that takes the Stripe session ID, validates it against Stripe, and returns only that booking. Mirrors what `verify-consultation-payment` does for consults. Patient self-views via the patient portal go through the regular own-only patient policy (which doesn't currently exist on this table — also add `Patients can view their own IV bookings` keyed on `patient_id`).

```sql
DROP POLICY IF EXISTS "Public can view by stripe session id"
  ON public.iv_drip_bookings;

CREATE POLICY "Patients can view their own IV bookings"
  ON public.iv_drip_bookings FOR SELECT
  USING (patient_id IN (SELECT id FROM patients WHERE user_id = auth.uid()));
```

Then update `IvLoungePaymentSuccess.tsx` (or wherever the unauthed post-checkout read happens) to invoke a new `verify-iv-payment` edge fn instead of querying the table directly.

### F-2 — `patients` intake-token SELECT leaks every active-token patient — CRITICAL

> **Status (2026-05-09):** Remediation written, awaiting Lovable apply.
> See `supabase/migrations/20260510030000_critical_rls_fixes.sql`. The
> migration drops the broken policy and adds
> `get_patient_by_intake_token(text)` SECURITY DEFINER RPC that
> filters on the actual token value. `src/pages/PublicIntake.tsx` is
> updated in the same change to call the RPC instead of the prior
> direct table query — the change ships together so the public intake
> page does not break when the migration is applied.

**File:** `supabase/migrations/20260202233922_625a89df-1859-4f76-9673-1db62adf6106.sql:4-7`

**Current:**
```sql
CREATE POLICY "Allow public intake token lookup"
ON public.patients FOR SELECT
USING (intake_token IS NOT NULL AND intake_token_expires_at > NOW());
```

**Problem:** The predicate doesn't check that the requested `intake_token` matches anything specific — it just checks that *some* token exists on the row and isn't expired. `SELECT * FROM patients WHERE intake_token IS NOT NULL` returns every patient with an active link. Patient names, emails, phones, DOBs, demographics all exposed.

**Recommended fix (R-2):**
Drop the policy and route the public intake page through an edge function `get-intake-by-token` that:
1. Takes the token in the request body
2. Uses service_role to look up exactly that one row
3. Returns only the fields the intake page needs (name, intake fields — NOT auth.user_id, stripe_customer_id, etc.)

```sql
DROP POLICY IF EXISTS "Allow public intake token lookup" ON public.patients;
```

Then update `PublicIntake.tsx` to call the new edge function. The function should refuse to return data if the token is malformed, expired, or already-consumed.

### F-3 — `user_roles` master-admin protection logic is inverted — CRITICAL

> **Status (2026-05-09):** Remediation written, awaiting Lovable apply.
> See `supabase/migrations/20260510030000_critical_rls_fixes.sql`. The
> migration drops the broken `Protect master admin role` policy and
> replaces it with a `RESTRICTIVE` DELETE policy that ANDs with the
> existing `Admins can manage all roles` permissive policy. Net
> effect: only admins can delete user_roles rows AND no one can
> delete the master admin row (rotating the master admin requires a
> deliberate SQL operator action, which is the appropriate friction).

**File:** `supabase/migrations/20251210233224_2ee946a8-0b0e-41c6-ab93-329b21bd1e06.sql:12-18`

**Current:**
```sql
CREATE POLICY "Protect master admin role"
ON public.user_roles FOR DELETE
USING (
  NOT (user_id = '31178dc3-3509-4cdd-8440-e75835bb1521' AND role = 'admin'::app_role)
  OR has_role(auth.uid(), 'admin'::app_role) = false
);
```

**Problem:** Trace for a non-admin authenticated user attempting to delete the master admin row:
- `NOT (true AND true)` → `false`
- `has_role(non-admin, 'admin') = false` → `true`
- `false OR true` → **TRUE → policy permits deletion**

The OR was intended to read "any caller can delete a non-master row" but the logic accidentally also reads "non-admin callers can delete the master row." Combined with Postgres RLS multi-policy OR semantics, a non-admin authenticated user can `DELETE FROM user_roles WHERE user_id = '31178...' AND role = 'admin'`, locking the clinic out of admin operations until the row is reseeded by a SQL operator.

**Recommended fix (R-3):**
```sql
DROP POLICY IF EXISTS "Protect master admin role" ON public.user_roles;

CREATE POLICY "Protect master admin role"
ON public.user_roles FOR DELETE
USING (
  has_role(auth.uid(), 'admin'::app_role)
  AND NOT (user_id = '31178dc3-3509-4cdd-8440-e75835bb1521' AND role = 'admin'::app_role)
);
```

This says: only admins can delete user_roles rows, and even admins cannot delete the master admin row. This intersects with the existing "Admins can manage all roles" ALL policy correctly (Postgres OR's policies, so the more permissive one wins for non-master rows; for the master row, neither policy permits delete).

**Severity:** Available exploit, but only authenticated users (not anonymous). Still launch-blocking because patient accounts are authenticated.


### F-8 — Staff vs admin DELETE separation is unenforced on `patients` — MEDIUM

> **Status (2026-05-09):** **Applied via Lovable migration `20260509071111`.**
> Cursor-authored migration at
> `supabase/migrations/20260510040000_rls_hardening_pass.sql`. The
> migration applies the split pattern to `patients`, `clinical_notes`,
> `lab_results`, `medications`, `treatment_plans`, `soap_notes`,
> `encounter_forms`, and `superbills`. For tables that previously had
> only the ALL policy as their DELETE permission, the migration adds an
> explicit `Admins can delete X` policy so dropping the ALL policy does
> not lock both staff and admins out of DELETE. Patients-only DELETE
> already existed via `Admins can delete patients` (created in
> 20251203022732). After this applies, only admins can DELETE on the
> eight tables.

**Problem:** The `Staff and admins can manage patients` ALL policy permits staff DELETE. The redundant `Admins can delete patients` policy doesn't constrain staff. Patient deletion is irreversible PHI loss; it should be admin-only.

**Recommended fix (R-8):** Replace the ALL policy with separate SELECT/INSERT/UPDATE policies. Keep the admin-only DELETE.

```sql
DROP POLICY IF EXISTS "Staff and admins can manage patients" ON public.patients;

CREATE POLICY "Staff and admins can read all patients" ON public.patients FOR SELECT
  USING (has_role(auth.uid(), 'admin'::app_role) OR has_role(auth.uid(), 'staff'::app_role));

CREATE POLICY "Staff and admins can update patients" ON public.patients FOR UPDATE
  USING (has_role(auth.uid(), 'admin'::app_role) OR has_role(auth.uid(), 'staff'::app_role))
  WITH CHECK (has_role(auth.uid(), 'admin'::app_role) OR has_role(auth.uid(), 'staff'::app_role));

CREATE POLICY "Staff and admins can insert patients" ON public.patients FOR INSERT
  WITH CHECK (has_role(auth.uid(), 'admin'::app_role) OR has_role(auth.uid(), 'staff'::app_role));

-- Existing "Admins can delete patients" policy is now the ONLY DELETE permission.
```

Apply the same split to `clinical_notes`, `lab_results`, `medications`, `treatment_plans`, `soap_notes`, `encounter_forms`, `superbills` if you want destructive ops admin-only on each. Decide per-table — clinical staff legitimately need to delete their own incorrect entries on some of these.

### F-9 — Open INSERT on `_payments` family — LOW/MEDIUM

These were intended for the legacy "verify-X-payment" flows that fell back to client-driven inserts. Today the writes happen via service role from the verify-X edge functions.

**Recommended fix (R-9):** Audit each table for active write paths. If only the service role writes, drop the public INSERT policy. If a client still writes for some reason, narrow the WITH CHECK predicate.
